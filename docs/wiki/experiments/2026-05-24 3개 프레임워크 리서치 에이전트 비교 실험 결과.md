# 실험: 3개 프레임워크 리서치 에이전트 비교

## 목표

동일한 리서치 에이전트 작업을 LangChain, LangGraph, Deep Agents로 각각 구현하고,
오케스트레이션 방식, 에이전트 루프 구조, 코드 복잡도를 비교한다.

## 설정

**작업:** "Python의 GIL(Global Interpreter Lock)이 무엇인지 설명하고, 이를 우회하는 방법을 정리해라."

**실행 환경:**
- Python 3.14.4
- langchain 1.3.1
- langgraph (최신)
- Mock LLM (FakeResearchLLM) — 실제 API 호출 없이 루프 구조 검증
- Mock search tool (고정 응답 반환)

**비교 기준:**
1. 코드 양 (Lines of Code — 주석·빈 줄 제외)
2. 오케스트레이션 방식 (명시적 그래프 vs 암묵적 루프)
3. 메시지 흐름 구조
4. 실행 시간 (런타임 초기화 포함)

**실험 코드 위치:**
- `examples/research_agent_comparison/`

## 메시지 흐름 (3개 모두 동일)

```
HumanMessage
  → AIMessage(tool_calls=[search(...)])   ← LLM 1번째 호출
  → ToolMessage(content="검색 결과")       ← ToolNode 실행
  → AIMessage(content="최종 답변")         ← LLM 2번째 호출
```

**검증됨:** 3개 프레임워크 모두 동일한 4-메시지 루프 패턴을 사용한다.

## 관찰 내용

### 1. 코드 복잡도 (LOC)

| API | LOC | 오케스트레이션 |
|-----|-----|--------------|
| `langchain.agents.create_agent` | 25 | 암묵적 (자동) |
| `langgraph.prebuilt.create_react_agent` | 24 | 암묵적 (자동) |
| `langgraph.graph.StateGraph` 수동 | 33 | **명시적** (노드+엣지) |

- `create_agent`와 `create_react_agent`는 거의 동등한 LOC
- StateGraph 수동 구성은 ~30% 더 많은 코드지만, 분기 로직이 코드에 드러남

### 2. 실행 시간

- **LangChain `create_agent`: 0.262s** — LangGraph 버전보다 **65배 느림**
- **LangGraph 두 버전: 0.004s** — 거의 동일

⚠️ **원인 가설:** `create_agent`는 내부적으로 `bind_tools()`를 매 호출마다 다시 실행하거나,
초기화 경로가 더 무거울 가능성이 있다. **Needs Source** — 정확한 원인 미확인.

### 3. API 파라미터 차이 (실제 확인됨)

| API | 시스템 프롬프트 파라미터 | tools |
|-----|----------------------|-------|
| `create_agent` | `system_prompt=` | `tools=` |
| `create_react_agent` | `prompt=` | `tools=` |

> 위키 실험 계획서의 `instructions=` 파라미터는 **잘못된 정보**였다.
> 실제 확인 결과: `create_agent` → `system_prompt=`, `create_react_agent` → `prompt=`.

### 4. bind_tools 동작

- `create_agent`는 **내부에서 자동으로 `model.bind_tools(tools)`를 호출**한다.
  - `factory.py` `_get_bound_model()` → `request.model.bind_tools(...)` 확인됨
  - 따라서 커스텀 LLM을 사용하려면 반드시 `bind_tools` 구현 필요
- `create_react_agent`와 LangGraph 수동은 `bind_tools` 없이도 동작 가능

### 5. Deep Agents 부재

- deepagents 패키지가 공개 PyPI에서 확인되지 않음
- 개념 스텁: `04_deep_agents_stub.py` 참고
- `create_agent` API가 `middleware=` 파라미터를 지원하므로, Deep Agents가 이 계층을 활용할 가능성이 높음 (⚠️ 가설)

## 핵심 정리

1. **메시지 흐름은 동일** — 어떤 고수준 API를 쓰든 `Human → tool_call → ToolMessage → final` 4단계
2. **create_agent vs create_react_agent** — 거의 동등, 파라미터명만 다름 (`system_prompt` vs `prompt`)
3. **StateGraph 수동** — 더 많은 코드이지만 분기 로직 제어권이 개발자에게 있음
4. **create_agent의 bind_tools 자동 호출** — 내부 동작 확인됨 (`factory.py` 참조)
5. **create_agent 초기화 오버헤드** — 0.262s vs 0.004s 차이, 원인 미확인
